8 /**
9  * @file include/gadc/gadc.h
10  *
11  * @addtogroup GADC
12  *
13  * @brief Module to abstract the very variable ADC interfaces of the underlying systems
14  *
15  * @details The reason why ChibiOS/GFX has it's own ADC abstraction is because
16  * the Chibi-OS drivers are very CPU specific and do not
17  * provide a way across all hardware platforms to create periodic
18  * ADC conversions. There are also issues with devices with different
19  * characteristics or periodic requirements on the same ADC
20  * device (but different channels). This layer attempts to solve these
21  * problems to provide a architecture neutral API. It also provides extra
22  * features such as multi-buffer chaining for high speed ADC sources.
23  * It provides one high speed virtual ADC device (eg a microphone) and
24  * numerous low speed (less than 100Hz) virtual ADC devices (eg dials,
25  * temperature sensors etc). The high speed device has timer based polling
26  * to ensure exact conversion periods and a buffer management system.
27  * The low speed devices are assumed to be non-critical timing devices
28  * and do not have any buffer management.
29  * Note that while only one high speed device has been provided it can
30  * be used to read multiple physical ADC channels on the one physical
31  * ADC device.
32  * All callback routines are thread based unlike the Chibi-OS interrupt based
33  * routines.
34  *
35  * @{
36  */

106 /**
107  * @brief Initialise the high speed ADC.
108  * @details Initialises but does not start the conversions.
109  *
110  * @param[in] physdev A value passed to describe which physical ADC devices/channels to use.
111  * @param[in] frequency The frequency to create ADC conversions
112  * @param[in] buffer The static buffer to put the ADC samples into.
113  * @param[in] bufcount The total number of conversions that will fit in the buffer.
114  * @param[in] samplesPerEvent The number of conversions to do before returning an event.
115  *
116  * @note If the high speed ADC is running it will be stopped. The Event subsystem is
117  * disconnected from the high speed ADC and any binary semaphore event is forgotten.
118  * @note bufcount must be greater than countPerEvent (usually 2 or more times) otherwise
119  * the buffer will be overwritten with new data while the application is still trying
120  * to process the old data.
121  * @note Due to a bug/feature in Chibi-OS countPerEvent must be even. If bufcount is not
122  * evenly divisable by countPerEvent, the remainder must also be even.
123  * @note The physdev parameter may be used to turn on more than one ADC channel.
124  * Each channel is then interleaved into the provided buffer. Note 'bufcount'
125  * and 'countPerEvent' parameters describe the number of conversions not the
126  * number of samples.
127  * As an example, if physdev turns on 2 devices then the buffer contains
128  * alternate device samples and the buffer must contain 2 * bufcount samples.
129  * The exact meaning of physdev is hardware dependent.
130  * @note The buffer is circular. When the end of the buffer is reached it will start
131  * putting data into the beginning of the buffer again.
132  * @note The event listener must process the event (and the data in it) before the
133  * next event occurs. If not, the following event will be lost.
134  * @note If bufcount is evenly divisable by countPerEvent, then every event will return
135  * countPerEvent conversions. If bufcount is not evenly divisable, it will return
136  * a block of samples containing less than countPerEvent samples when it reaches the
137  * end of the buffer.
138  * @note While the high speed ADC is running, low speed conversions can only occur at
139  * the frequency of the high speed events. Thus if high speed events are
140  * being created at 50Hz (eg countPerEvent = 100, frequency = 5kHz) then the maximum
141  * frequency for low speed conversions will be 50Hz.
142  *
143  * @api
144  */
145 void gadcHighSpeedInit(uint32_t physdev, uint32_t frequency, adcsample_t *buffer, size_t bufcount, size_t samplesPerEvent);

210 /**
211  * @brief Perform a single low speed ADC conversion
212  * @details Blocks until the conversion is complete
213  * @pre This should not be called from within a GTimer callback as this routine
214  * blocks until the conversion is ready.
215  *
216  * @param[in] physdev A value passed to describe which physical ADC devices/channels to use.
217  * @param[in] buffer The static buffer to put the ADC samples into.
218  *
219  * @note This may take a while to complete if the high speed ADC is running as the
220  * conversion is interleaved with the high speed ADC conversions on a buffer
221  * completion.
222  * @note The result buffer must be large enough to store one sample per device
223  * described by the 'physdev' parameter.
224  * @note If calling this routine would exceed @p GADC_MAX_LOWSPEED_DEVICES simultaneous low
225  * speed devices, the routine will wait for an available slot to complete the
226  * conversion.
227  * @note Specifying more than one device in physdev is possible but discouraged as the
228  * calculations to ensure the high speed ADC correctness will be incorrect. Symptoms
229  * from over-running the high speed ADC include high speed samples being lost.
230  *
231  * @api
232  */
233 void gadcLowSpeedGet(uint32_t physdev, adcsample_t *buffer);
